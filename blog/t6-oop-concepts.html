Inheritance Vs. Composition

Feb. 21, 2015

In the example above the Class child inherits the class Parent. As a consequence even though there is no method implicit() in child when you run the code it will return "PARENT implicit()" twice.

Were you to have another implicit() method written in the child class it would replace any inherited methods. If you wanted your new method to run but also wanted the original method with the same name to run, you can use the Ruby built in function super()

The most common use of super() is in the initialize method of a child class where you usually want to also initialize whatever the parent class needs.

You can get into trouble using inheritance, especially once you start chaining many classes with inheritance. Instead of using inheritance we can instead use composition. For example:

Here we have an instance of the class Other at the initialization of Child. This way we guarantee that every method of Child will have access to the methods of the Other class through the instance variable @other.

Another way to go by this is by using modules. For example:

In this example we see that modules are a different type of object and start with the module module_name syntax. By using the include module_name in a class we give it access to the methods of the module. This is an easy way to bring in methods that are repeated often in your program.

It is often tempting to always use inheritance as it is a fast and easy way to drag methods from another class. This can start to become problematic when chains of classes are used with inheritance. When possible it is best practice to use composition and modules.
